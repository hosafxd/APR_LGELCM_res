# CC_CRITIQUE — Radiology Report Evaluation
---

## 1. DATA STRUCTURE (WHAT EACH FILE MEANS)

* **RESULT_OF_16/info.docx**
  → Clinical similarity assessment created by Claude Web LLM
  → Claude read both the ground truth and generated reports for all 16 pairs
  → Estimated clinical similarity as a percentage range (e.g. 10–15%, 70–75%)
  → Also identified specific missed findings, hallucinations, and failure patterns per report

* **RESULT_OF_16/results/**
  → Entity matching pipeline output (Colab)
  → Same 16 reports evaluated with automated entity-level F1 scoring
  → Produces: Observation F1, Location F1, Degree F1, Trend F1, MACRO_FL

* **RESULT_OF_TEST / TEST_2 / TEST_3**
  → Synthetic test sets — reports generated by LLM from GT files with similarity prompts
  → TEST_2: prompt requested 30–40% similarity → low scores expected
  → TEST / TEST_3: prompt requested 80–90% similarity → high scores expected

* **All results folders**
  → Same pipeline code, same evaluation logic, different datasets

## 2. CLINICAL ASSESSMENT vs PIPELINE (RESULT_OF_16)

### What each system measures

* **Clinical LLM (info.docx):** Reads both reports and estimates how much of the clinical content matches. Sensitive to whether the correct findings are named correctly, even without structured matching.

* **Pipeline (MACRO_FL):** Counts matched medical entities (observations, locations, degrees, trends) using structured F1. Final score = average F1 across active fields × coverage factor.

### Scores are not directly comparable

* Clinical LLM mean: ~22% (midpoint average across 16 reports)
* Pipeline MACRO_FL mean: 0.1333

These numbers are on different scales and use different logic. What matters is whether they agree on which reports are better or worse.

### Where they agree

* valid_1005_b_1 / valid_1005_b_2 → Both systems identify these as among the worst reports.
  (Clinical: 5–10%. GT contained hemorrhagic pericardial effusion and surgical materials; model generated entirely unrelated template findings.)

* valid_1002_a_x → Both place these in the lower-middle range (~15–20% clinical, ~16% pipeline).

* valid_1005_a_1 → Both rate this as one of the better performing reports in the dataset.

### Where they strongly disagree

**valid_1000_a_2 — Clinical rank 1, Pipeline rank 15**
The GT is a normal CT and the model also produced a normal CT. Clinical assessment: 85–90%. The model got the right answer. Pipeline MACRO_FL: 8.9%.
→ Why the pipeline fails here: a normal report contains few positive entities. The coverage factor, which multiplies F1 by entity density, is inherently low for normal cases. The pipeline cannot proportionally reward correctly identified absence of pathology.

**valid_1001_a_1 — Clinical rank 2, Pipeline rank 14**
GT is mostly normal with millimetric nodules; model produced a fully normal report. Clinical: 70–75% (most content correct, just missed small nodules). Pipeline: 9.1%.
→ Why: Degree F1 = 0.0 (no degree terms matched at all). This collapses fl_mean to 0.5, and multiplied by low coverage gives MACRO_FL = 9.1%. A single zero-F1 field has a disproportionate effect on the final score.

**valid_1003_a_1 — Clinical rank 3, Pipeline rank 16 (last)**
Clinical: 40–50% (partial structural match, agreement on no mass/infiltration). Pipeline: 3.3%.
→ Why: despite partial clinical agreement, the specific entity content barely matched. Pipeline entity matching is stricter than clinical impression agreement.

### Conclusion for Section 2

The two systems diverge most on two cases:
1. **Normal reports** — pipeline undercounts because entity density is low
2. **Reports with zero-F1 fields** — a single field with no matches collapses the pipeline score regardless of other field performance

For complex pathological cases, the pipeline alignment with clinical judgment is reasonably good. The pipeline is most reliable as a discriminator when reports contain many varied entities (complex pathology). It is less reliable for simple or normal reports.

## 3. MODEL BEHAVIOR FINDINGS (FROM info.docx)

This section summarizes what the clinical reading of the 16 reports revealed about the model's generation behavior. These are model-level observations, not pipeline problems.

### The model uses a limited template set

Across nearly all generated reports, the model's output consists of a small fixed vocabulary of findings:
- emphysema
- atelectasis
- nodules
- atherosclerosis
- hiatal hernia
- OR: fully normal report

When the ground truth falls within this template set, the model performs well. When the ground truth contains findings outside this set, the model either misses them entirely or substitutes template findings.

### Two failure modes

**Mode 1 — GT is pathological, model says normal or uses template:**
Examples: valid_1005_a_2 (GT: pericardial calcifications, pleural effusion, atelectasis, peribronchial thickening, emphysema → Generated: completely normal), valid_1004_a_1 and valid_1004_a_2 (GT: paracardiac cystic lesion, LAP, consolidation → Generated: completely normal), valid_1005_b_1/b_2 (GT: hemorrhagic pericardial effusion, surgical materials → Generated: emphysema, nodules, unrelated findings).

**Mode 2 — GT is normal, model adds false pathology:**
Examples: valid_1000_a_1 (GT: normal CT → Generated: adds emphysema, atelectasis, nodules, atherosclerosis, hiatal hernia). Clinical similarity: 30–35%, despite a normal GT.

**Mode 3 — GT is normal, model also normal (success case):**
valid_1000_a_2: GT normal CT, Generated normal CT → Clinical similarity: 85–90%.

### Specific missed finding categories

Based on info.docx across the 16 reports:
* **Pericardial findings** (calcifications, effusions, surgical changes) — consistently missed
* **Pleural pathology** (effusions, thickening) — frequently missed in complex cases
* **Vascular findings** (pulmonary artery dilatation, aortic changes, pacemaker) — missed
* **LAP (lymphadenopathy)** — inconsistently captured
* **Thymic remnants** — missed
* **Findings in upper abdomen** (liver disease, cholecystectomy) — missed when present

### Summary

The model's clinical accuracy is low for complex or atypical pathology. Performance is highest when the case is normal or when pathology falls within the model's template vocabulary. This pattern explains why most reports scored 10–25% in clinical assessment — the model correctly handles structural negatives but misses or fabricates specific pathological findings.

## 4. CROSS-DATASET PROBLEM (INCONSISTENT PIPELINE RESULTS)

Two synthetic datasets used the same generation prompt ("80–90% similar"), but pipeline results are very different:

| Metric      | TEST   | TEST_3 |
| ----------- | ------ | ------ |
| ROUGE-1     | 0.495  | 0.919  |
| Obs F1      | 0.421  | 0.851  |
| Location F1 | 0.286  | 0.591  |
| Degree F1   | 0.160  | 0.783  |
| MACRO_FL    | 0.184  | 0.419  |

Scores are approximately 2× higher in TEST_3. Both datasets used the same pipeline code.

### Why?

The pipeline is sensitive to report structure. TEST_3 reports are structurally simpler and closer to the template the LLM used when paraphrasing. TEST reports are structurally harder — the LLM could not achieve 80–90% actual semantic similarity despite the prompt requesting it.

This means the prompt-level similarity claim is not reliable as ground truth. The pipeline correctly detects the actual structural divergence even when the generation prompt specified high similarity.

---

### LOCATION PROBLEM

* TEST: many false positives in location matching
* TEST_3: more correct location matches

Reason: different granularity in describing the same location.
→ "bilateral lungs" vs "right lung" + "left lung" — clinically identical, but pipeline treats as different entities

This is not a model failure. It is a normalization gap in the pipeline.

✔ Fix: merge "right X" + "left X" → "bilateral X" before matching

---

### DEGREE PROBLEM

* TEST: Degree F1 = 0.160 (very low)
* TEST_3: Degree F1 = 0.783 (high)

Reason: degree vocabulary varies significantly across reports. Terms like "mild", "minimal", "slight", "subtle" are near-synonyms but may not be linked in the embedding synonym cache if those specific terms were not present when the cache was built.

If a degree term is missing from embedding_synonyms.json, the matching score for that term = 0.0, even if it is clinically equivalent to a matched term.

This explains the degree collapse in TEST but not TEST_3 — the TEST reports contained degree terms that were underrepresented in the synonym cache.

✔ Fix: run the tokenizer + embedding steps on each new test dataset before evaluation to ensure all new terms are added to entity.json and embedding_synonyms.json

---

### OBSERVATION FIELD

* More stable across datasets than location or degree
* Still affected by extra generated entities

Problem: the model sometimes generates more entities than the ground truth. Even if the extra entities are clinically plausible, the pipeline counts them as false positives, reducing precision.

✔ Fix: consider recall-weighted metric for cases where generated output is more verbose than GT

---

### TREND FIELD

* Almost always zero in chest CT reports
* Included in the MACRO_FL average
* Because Trend F1 = 0 for nearly all reports, it pulls the average down
* This creates a structural ceiling on the maximum achievable score (~0.75 effective maximum when trend is consistently zero)

✔ Fix: exclude fields where both GT and prediction have zero entities from the field average

## 5. VALIDATION (DOES THE PIPELINE WORK?)

Pipeline MACRO_FL across all four datasets:

| Dataset | Similarity Target | MACRO_FL |
|---|---|---|
| RESULT_OF_TEST_2 | 30–40% (low) | 0.0935 |
| RESULT_OF_TEST | 80–90% (high) | 0.1841 |
| RESULT_OF_TEST_3 | 80–90% (high) | 0.4191 |
| RESULT_OF_16 | Real model output | 0.1333 |

The pipeline correctly orders the datasets: TEST_2 (low similarity prompt) < real model < TEST (high similarity prompt) < TEST_3 (high similarity prompt, simpler structure).

The real model scores between 30–40% and 80–90% synthetic similarity, which is consistent with clinical assessment showing mostly 10–25% similarity with occasional high performers.

**Important caveat:** TEST and TEST_3 used the same 80–90% similarity prompt, yet their MACRO_FL values differ by 2.3×. This shows the pipeline score is not solely determined by intended similarity — report structure, entity vocabulary, and LLM paraphrasing quality all influence the final number. The pipeline correctly measures actual entity overlap, not promised similarity.

## 6. MAIN PROBLEMS (ROOT CAUSES)

### A) Location mismatch

* Different granularity levels for the same anatomical location
* "bilateral lungs" ≠ "right lung" + "left lung" in the current pipeline
* Causes false positives and false negatives in Location F1

✔ Fix: pre-matching normalization — convert "right X" + "left X" → "bilateral X"


---

### C) Too many predicted entities

* Model generates more entities than ground truth (verbosity)
* Pipeline penalizes extra entities as false positives
* Even if the additional entities are clinically appropriate, precision drops

✔ Fix: consider recall-weighted or recall-primary metric for verbose outputs

---

### D) Trend field dead weight

* Trend entities are almost never present in chest CT reports
* Including trend in the MACRO_FL average caps the maximum achievable score
* Effective ceiling: ~0.75 (when trend is consistently zero, only 3 of 4 fields contribute)

✔ Fix: compute field average only over fields where GT + prediction entity count > 0

---

### E) One-to-many matching without global optimality

* Current matching: for each GT entity, finds all pred groups with soft match, scores them, picks the best (local greedy selection)
* There is no consumed/exclusivity tracking — the same pred group can be matched by multiple GT entities
* This is effectively one-to-many at the group level, not one-to-one
* Global optimal assignment is not computed

✔ Fix: Hungarian algorithm (scipy.optimize.linear_sum_assignment) — provides one-to-one exclusivity + global optimal assignment

---

### F) Pipeline blind spot for correctly-normal reports

* When both GT and generated report describe a normal CT, entity count is low
* Coverage factor (based on entity density) remains low
* MACRO_FL underscores these reports despite the model being clinically correct
* valid_1000_a_2 is the clearest example: clinical similarity 85–90%, MACRO_FL 8.9%

✔ Fix: introduce a separate scoring track or bonus for correctly-matched absence-dominant reports; or report Pathological F1 and Normal F1 separately

---

## 7. FINAL ASSESSMENT

The pipeline idea is correct:
* Entity-based evaluation is more meaningful than pure text similarity
* It measures whether the right medical findings were identified, not just whether the same words appear

The pipeline is reliable as a **relative discriminator** — it correctly separates clearly good reports from clearly bad ones, and correctly orders datasets by similarity level.

The pipeline is not yet reliable as an **absolute ruler**:
* Normal reports are underscored (coverage blind spot)
* A single zero-F1 field disproportionately collapses the final score
* Cross-dataset variation is high even for the same similarity target

### Summary of problems by category

**Category 1 — Structural design problems (affect all datasets equally):**
* Trend field dead weight → caps maximum score
* No bilateral location normalization → location F1 underestimated
* One-to-many matching → inflates TP count inconsistently
* Coverage factor penalizes normal reports

**Category 2 — Data sensitivity problems (vary by dataset):**
* Degree synonym cache not updated per dataset → degree collapse on new data
* Verbose generation → false positive inflation in observation/location

**Category 3 — Model behavior problems (not a pipeline problem, but affects interpretation):**
* Limited template vocabulary → hallucination in complex cases
* Two distinct failure modes: false negatives (misses pathology) and false positives (adds template findings to normal GT)
* Performance drops dramatically when GT contains findings outside the model's template set

### Final conclusion

The system works as a compass, not a ruler:
* It correctly identifies direction (better vs worse)
* It correctly separates datasets by similarity level
* But absolute scores shift based on report structure, entity vocabulary, and field coverage

Fixing the structural issues (trend weighting, bilateral normalization, coverage scoring for normal reports, Hungarian matching) will increase both absolute score reliability and cross-report consistency.
